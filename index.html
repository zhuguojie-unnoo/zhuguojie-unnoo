聊聊弹框那些事儿

UIAlertView、UIAlertController

苹果官方 API 中建议开发者在 iOS8 及其以后的系统版本中使用 UIAlertController。接下来的篇幅将主要讨论 UIAlertController 使用时的注意点、苹果为这样调整的原因和这个调整引发的思考。

1、UIAlertController 使用时的注意点

对于如何使用 UIAlertController 这个不过过多的介绍，它实质就是一个控制器，只需像对待其他控制器一样，把他弹出来便可。至于设置参数和事件这里不做描述，看一下 API 里面的方法和属性便知。

[self presentViewController:alertViewController animated:YES completion:nil];

当将它的 preferredStyle 设置为 UIAlertControllerStyleActionSheet 的时，在 iPad（iPhone 中正常） 中一定要设置 ActionSheet 的弹出位置。否则会报错，控制台输出的错误信息如下：

 *** Terminating app due to uncaught exception 'NSGenericException', reason: 'Your application has presented a UIAlertController () of style UIAlertControllerStyleActionSheet. The modalPresentationStyle of a UIAlertController with this style is UIModalPresentationPopover. You must provide location information for this popover through the alert controller's popoverPresentationController. You must provide either a sourceView and sourceRect or a barButtonItem.  If this information is not known when you present the alert controller, you may provide it in the UIPopoverPresentationControllerDelegate method -prepareForPopoverPresentation.'

设置 ActionSheet 的位置

    if ([[UIDevice currentDevice] userInterfaceIdiom] == UIUserInterfaceIdiomPad) {
        UIPopoverPresentationController *popover = controller.popoverPresentationController;
        popover.permittedArrowDirections = 0;
        popover.sourceView = controller.view;
        popover.sourceRect = (CGRect){SCREEN_WIDTH / 2, SCREEN_HEIGHT / 2, 0, 0};
    }

2、苹果为这样调整的原因

要说出苹果修改的真实意图，目前我的道行还够，不过通过在使用过程中的一些体验和经验，这里简单的分析一下。

a、简化代码、降低维护成本

iOS8 以前使用弹出式对话框时需要使用 UIAlertView 和 UIActionSheet，当需要互相切换他们的样式的时候，非常麻烦，不亚于重写。但是如果使用 UIAlertController 只需要改变 preferredStyle 参数的值变可以做到。

b、简化逻辑

iOS8 以前我们需要设置条目的事件时，需要设置代理并在回调中判断其 index 或者是一些我们设置的标识，其次如果界面有多个弹框时，还需要判断是那个弹框传来的事件。但是使用 UIAlertController 后，条目的事件直接写到该条目的 Block 中，因此省去不少逻辑判断代码。

3、思考

我不知你们曾经在设计一些自定义弹框或者一些类似弹框效果的控件时是有这样设计过：使用 UIWindow 然后将相关控件添加到上面，使用的时候直接调用一个方法显示出来。我之前有这么做过。但是在使用 UIAlertController 后，我觉得我们应该调整一下了。为什么不使用 UIAlertController 的设计思路呢？如果使用了控制器，其实会给我们省去很多不必要的问题，比如， window 的 level 问题，输入框键盘的window 和当前 window 的关系问题，生命周期的问题。
另外，方便设计，比如你需要一个背景是半透明的，点击背景是消失，只需要创建一个父类，把这个功能做进去，然后其他的继承于他便可。

IOS图片圆角效果优化

显示为圆角的图片现在很常见，之前使用的时候一直没有考虑过效率问题，因为系统提供了属性，直接设置就完事了。（ps：在优化前我一直是使用这种方法来实现的，简单快捷）导致性能不好的原因主要是引起了 offscreen rendering。

造成 offscreen rendering的原因有：

* Any layer with a mask (layer.mask)
* Any layer with layer.masksToBounds being true
* Any layer with layer.allowsGroupOpacity set to YES and layer.opacity is less than 1.0
* Any layer with a drop shadow (layer.shadow*).
* Any layer with layer.shouldRasterize being true
* Any layer with layer.cornerRadius, layer.edgeAntialiasingMask, layer.allowsEdgeAntialiasing
优化的过程中尽量不要使用上面的属性或者方法。 



1、使用系统的属性
设置圆角的半径，设置超出部分剪裁。


imageView.layer.cornerRadius = 20;
imageView.masksToBounds = YES;




当然也有说设置 shouldRasterize 为YES,但是如果是在一个界面有“大量”的圆角图片的 tableView 或者 conllectionView 中时，是没有效果的。


2、使用贝塞尔曲线画出 imageView
在分类中写一个方法，使用的时候直接调用该方法。


    self.clipsToBounds = YES;
    UIBezierPath *path = [UIBezierPath bezierPathWithRoundedRect:CGRectMake(0, 0, radius * 2, radius * 2) cornerRadius:radius];
    CAShapeLayer *maskLayer = [CAShapeLayer layer];
    maskLayer.path = path.CGPath;
    self.layer.mask = maskLayer;




3、处理成圆角图片
在图片显示到界面上时，先把图片处理成员圆角图片。大方向有两种方案：服务器处理和客户端处理。服务器返回圆角图片在此我就不多说了，使用的是不会有性能问题的。我们来说说客户端将图片处理成圆角的方案。


a.从网络上拿到图片时将图片处理成圆角存放在本地
b.从网络上拿到图片存放在本地是原图，加载到内存或缓存时处理成圆角
QQ 采用就是从服务器获取存在本地是原图，显示的时候是圆角图片的方案的。
  优点：界面不卡顿、代码量一般


  缺点：此方法比较消耗 CPU 资源 ，代码需要时刻维护（如结合第三框架来修改，需要随时维护处理图片的代码）


Tip: 以上两种处理图片的方法可以结合 SDWebImage 的第三方框架来实现，在对应的位置去处理图片就行。


UIGraphicsBeginImageContextWithOptions(avatarImageView.bounds.size, NO, [UIScreen mainScreen].scale);
[[UIBezierPath bezierPathWithRoundedRect:avatarImageView.bounds
                              cornerRadius:50] addClip];
[image drawInRect:avatarImageView.bounds];
avatarImageView.image = UIGraphicsGetImageFromCurrentImageContext();
UIGraphicsEndImageContext();





4、在显示图片的控件上加上一个圆角遮罩
  优点：性能好，不卡顿，代码量比较少


Tip: 需要注意圆角遮罩的颜色，必须与背景色相同，比如和 cell 的背景色相同。点击状态和非点击状态需要变换颜色与其保持一致。


总结
   优化是权衡各种方案的利弊，然后结合自己的实际情况，选择对于自己项目来说最优的方案。


比如：界面上圆角图片比较少的情况（只有几个）那么直接使用第一种或者第二种都行，这几张图片的消耗不足于引起界面的卡顿。
